Transparency — Catalogue update commitment

App: Swiss Citizenship Test · Developer: Vincenzo Casini · Last updated: 11 May 2026

This application is a study companion, not an official preparation tool. For the actual naturalization test, candidates must always refer to the official platform of their canton, linked in the table below.

Why this page exists

This app sources its question pools from official cantonal websites for 8 Swiss cantons. Because each canton publishes its catalogue differently and updates it on its own schedule, users and institutions need a way to verify how this app keeps its data in sync with official sources. This page documents the verification process, the declared service level, the reporting channel for discrepancies, and the audit trail available on request.

This commitment is voluntary. The app is not affiliated with any government authority. It is independently developed by a naturalization applicant.

Service Level — what we promise

Source verification: all 8 official cantonal sources are checked once every 24 hours via an automated continuous-integration pipeline. Checks run on a staggered schedule (between 03:00 and 07:00 UTC) to avoid concurrent writes to the audit log.

Drift detection latency: a change in any official source is detected within 24 hours of publication.

Standard refresh window: when a change is detected, the updated catalogue is published in a new app version within 7 days.

Emergency refresh window: critical issues (factual errors flagged by users or canton authorities) are addressed within 72 hours.

Metadata review: the legal basis and procedure scope for each canton is manually re-verified every 6 months or on-demand if the canton announces a procedural change.

How verification works

The pipeline is built in four layers:

  1. Scraping — automated parsers read the official PDF or API for each canton, normalize the question and answer content, and produce a structured snapshot.
  2. Refresh orchestration — when a refresh is triggered, the parser re-runs from scratch. Existing question IDs are preserved; new questions get fresh IDs; questions removed by the canton are marked deprecated (never silently deleted).
  3. Drift detection — a nightly job computes a content hash of every official source and compares it against the previous baseline. If the hash differs, the job flags a drift event.
  4. Audit log — every verification run, drift or not, is recorded in an append-only structured log (one JSONL file per canton). Each entry includes timestamp, source URL, response status, content hash, previous hash, and any detected changes.
  5. No auto-acknowledgement — a drift event stays flagged as unresolved until a human reviews the change, updates the question set if needed, and commits a new baseline hash. The pipeline never silently resets the drift flag on its own.

Per-question provenance

Every question in the app carries structured metadata that records:

  • Extraction method (PDF text, OCR, manual curation, or API)
  • Extraction date and the hash of the source document at extraction time
  • Original answer format (true-false, MCQ, open) and the format actually delivered in the app
  • An adapted flag set when the delivered format differs from the original (for example, open questions reformatted as MCQ with alternative choices)

This metadata makes every question independently verifiable against its source.

Reporting discrepancies

If you find a question that is outdated, incorrect, or missing relative to its official source, please report it. Reports are processed within the SLA above — 72 hours for critical issues, 7 days for standard updates.

Audit log access

Every verification run is recorded in append-only structured logs (one JSONL file per canton). Logs include timestamp, source URL, response status, content hash, and any detected changes. Logs are kept indefinitely on a dedicated git branch.

The audit log is not published openly because it lives in the application's production repository. It is available for inspection by cantonal naturalization authorities, the State Secretariat for Migration, accredited researchers, or journalists on written request at swisscitizenshiptest@gmail.com. Inspection requests are honoured within 5 working days.

Versioning policy

Each catalogue snapshot is identified by:

  • An ISO date (when the snapshot was captured from the official source)
  • A content hash (computed over the normalized catalogue)
  • An app version (the first published app release that ships this snapshot)

Historical snapshots are preserved so any past version can be inspected and compared against the live one.

Comment fonctionne la vérification

Le pipeline est construit en quatre couches :

  1. Extraction (scraping) — des parseurs automatisés lisent le PDF ou l'API officielle de chaque canton, normalisent le contenu des questions et des réponses, et produisent un instantané structuré.
  2. Orchestration des rafraîchissements — lorsqu'un rafraîchissement est déclenché, le parseur est ré-exécuté depuis zéro. Les identifiants des questions existantes sont préservés ; les nouvelles questions reçoivent des identifiants frais ; les questions retirées par le canton sont marquées comme dépréciées (jamais supprimées silencieusement).
  3. Détection des écarts — un job nocturne calcule une empreinte (hash) du contenu de chaque source officielle et la compare à la référence précédente. Si l'empreinte diffère, le job signale un événement de dérive.
  4. Journal d'audit — chaque exécution de vérification, qu'il y ait dérive ou non, est enregistrée dans un journal structuré en mode ajout seul (un fichier JSONL par canton). Chaque entrée contient un horodatage, l'URL source, le code de réponse, l'empreinte du contenu, l'empreinte précédente et tout changement détecté.
  5. Pas d'acquittement automatique — un événement de dérive reste signalé comme non résolu jusqu'à ce qu'un humain examine le changement, mette à jour le jeu de questions si nécessaire, et valide un nouvel hash de référence. Le pipeline ne réinitialise jamais silencieusement l'indicateur de dérive de lui-même.

Diese App bezieht ihre Fragenpools von den offiziellen kantonalen Websites von 8 Schweizer Kantonen. Service-Level — was wir zusichern

Quellenverifizierung: alle 8 offiziellen kantonalen Quellen werden alle 24 Stunden über eine automatisierte Continuous-Integration-Pipeline geprüft. Die Prüfungen erfolgen gestaffelt (zwischen 03:00 und 07:00 UTC), um gleichzeitige Schreibvorgänge im Audit-Log zu vermeiden.

Wie die Verifizierung funktioniert

Die Pipeline ist in vier Schichten aufgebaut:

  1. Scraping — automatisierte Parser lesen die offizielle PDF oder API jedes Kantons, normalisieren den Inhalt von Fragen und Antworten und erzeugen einen strukturierten Snapshot.
  2. Aktualisierungsorchestrierung — wird eine Aktualisierung ausgelöst, läuft der Parser von Grund auf neu. Bestehende Fragen-IDs werden beibehalten; neue Fragen erhalten frische IDs; vom Kanton entfernte Fragen werden als veraltet markiert (niemals stillschweigend gelöscht).
  3. Änderungserkennung — ein nächtlicher Job berechnet einen Inhalts-Hash jeder offiziellen Quelle und vergleicht ihn mit der vorherigen Referenz. Weicht der Hash ab, meldet der Job ein Drift-Ereignis.
  4. Audit-Log — jede Verifizierungsausführung, ob mit oder ohne Drift, wird in einem strukturierten, nur ergänzenden Log (eine JSONL-Datei pro Kanton) festgehalten. Jeder Eintrag enthält Zeitstempel, Quell-URL, Antwortcode, Inhalts-Hash, vorheriger Hash und allfällige erkannte Änderungen.
  5. Keine automatische Quittierung — ein Drift-Ereignis bleibt als ungelöst markiert, bis ein Mensch die Änderung überprüft, den Fragensatz bei Bedarf aktualisiert und einen neuen Referenz-Hash eincheckt. Die Pipeline setzt die Drift-Markierung niemals stillschweigend von selbst zurück.

Trasparenza — Impegno di aggiornamento del catalogo

App: Swiss Citizenship Test · Sviluppatore: Vincenzo Casini · Ultimo aggiornamento: 11 maggio 2026

Questa app è un compagno di studio, non uno strumento ufficiale di preparazione. Per il test ufficiale di naturalizzazione, i candidati devono sempre fare riferimento alla piattaforma ufficiale del loro cantone, indicata nella tabella sottostante.

Perché esiste questa pagina

Questa app attinge i suoi pool di domande dai siti cantonali ufficiali di 8 cantoni svizzeri. Poiché ogni cantone pubblica il proprio catalogo in modo diverso e lo aggiorna secondo il proprio calendario, gli utenti e le istituzioni hanno bisogno di un modo per verificare come questa app mantenga i suoi dati allineati alle fonti ufficiali. Questa pagina documenta il processo di verifica, il livello di servizio dichiarato, il canale di segnalazione delle discrepanze e il log di audit disponibile su richiesta.

Questo impegno è volontario. L'app non è affiliata ad alcuna autorità governativa. È sviluppata in modo indipendente da un candidato alla naturalizzazione.

Livello di servizio — i nostri impegni

Verifica delle fonti: tutte le 8 fonti cantonali ufficiali vengono controllate ogni 24 ore tramite una pipeline automatizzata di integrazione continua. I controlli avvengono in modo scaglionato (tra le 03:00 e le 07:00 UTC) per evitare scritture concorrenti sul log di audit.

Latenza di rilevamento delle modifiche: una modifica di una fonte ufficiale viene rilevata entro 24 ore dalla pubblicazione.

Finestra standard di aggiornamento: quando viene rilevata una modifica, il catalogo aggiornato viene pubblicato in una nuova versione dell'app entro 7 giorni.

Finestra di emergenza: i problemi critici (errori fattuali segnalati da utenti o autorità cantonali) vengono affrontati entro 72 ore.

Revisione dei metadati: la base legale e il perimetro procedurale di ogni cantone vengono verificati manualmente ogni 6 mesi o su richiesta se il cantone annuncia un cambiamento procedurale.

Come funziona la verifica

La pipeline è costruita in quattro livelli:

  1. Scraping — parser automatizzati leggono il PDF o l'API ufficiale di ogni cantone, normalizzano il contenuto di domande e risposte e producono uno snapshot strutturato.
  2. Orchestrazione degli aggiornamenti — quando viene attivato un aggiornamento, il parser viene rieseguito da zero. Gli ID delle domande esistenti vengono preservati; le nuove domande ricevono ID nuovi; le domande rimosse dal cantone vengono marcate come deprecate (mai cancellate silenziosamente).
  3. Rilevamento delle modifiche — un job notturno calcola un hash del contenuto di ogni fonte ufficiale e lo confronta con la baseline precedente. Se l'hash differisce, il job segnala un evento di drift.
  4. Log di audit — ogni esecuzione di verifica, con o senza drift, viene registrata in un log strutturato in sola aggiunta (un file JSONL per cantone). Ogni voce contiene timestamp, URL fonte, codice di risposta, hash del contenuto, hash precedente ed eventuali modifiche rilevate.
  5. Nessuna conferma automatica — un evento di drift rimane segnalato come non risolto finché un essere umano non esamina la modifica, aggiorna il set di domande se necessario e fa il commit di un nuovo hash di riferimento. La pipeline non azzera mai silenziosamente l'indicatore di drift da sola.

Provenienza per domanda

Ogni domanda dell'app porta con sé metadati strutturati che registrano:

  • Metodo di estrazione (testo PDF, OCR, curazione manuale o API)
  • Data di estrazione e hash del documento sorgente al momento dell'estrazione
  • Formato originale della risposta (vero-falso, MCQ, aperta) e formato effettivamente erogato in app
  • Un flag adattato impostato quando il formato erogato differisce dall'originale (ad esempio, domande aperte riformattate come MCQ con scelte alternative)

Questi metadati rendono ogni domanda verificabile indipendentemente rispetto alla sua fonte.

Segnalazione di discrepanze

Se trovi una domanda obsoleta, errata o mancante rispetto alla sua fonte ufficiale, segnalala. Le segnalazioni vengono trattate nei tempi dichiarati sopra — 72 ore per i problemi critici, 7 giorni per gli aggiornamenti standard.

  • Email: swisscitizenshiptest@gmail.com

Accesso al log di audit

Ogni esecuzione di verifica è registrata in log strutturati in sola aggiunta (un file JSONL per cantone). I log contengono timestamp, URL fonte, codice di risposta, hash del contenuto ed eventuali modifiche rilevate. I log sono conservati a tempo indeterminato su un branch git dedicato.

Il log di audit non è pubblicato apertamente perché risiede nel repository di produzione dell'applicazione. È disponibile per l'ispezione da parte delle autorità cantonali di naturalizzazione, della Segreteria di Stato della migrazione, di ricercatori accreditati o di giornalisti, su richiesta scritta a swisscitizenshiptest@gmail.com. Le richieste di ispezione sono evase entro 5 giorni lavorativi.

Politica di versionamento

Ogni snapshot di catalogo è identificato da:

  • Una data ISO (quando lo snapshot è stato catturato dalla fonte ufficiale)
  • Un hash di contenuto (calcolato sul catalogo normalizzato)
  • Una versione dell'app (la prima release pubblicata che spedisce questo snapshot)

Gli snapshot storici sono preservati in modo che qualsiasi versione passata possa essere ispezionata e confrontata con la versione attualmente in vigore.

Informazioni su questa app

Swiss Citizenship Test è uno strumento di studio gratuito, senza pubblicità e senza account per il test di conoscenze della naturalizzazione svizzera. È sviluppato in modo indipendente da un candidato alla naturalizzazione residente in Svizzera. Non è affiliato ad alcuna autorità governativa. Il suo scopo è integrare gli strumenti ufficiali di preparazione — non sostituirli.

Per domande su questo impegno di trasparenza o per discutere una collaborazione istituzionale: swisscitizenshiptest@gmail.com.

Nota per le istituzioni: se rappresenti un ufficio cantonale o comunale di naturalizzazione e vuoi verificare il log di audit o discutere di come questa app fa riferimento al catalogo del tuo cantone, contattaci all'indirizzo sopra. Siamo lieti di dimostrare la pipeline in diretta, condividere il log di audit per il tuo cantone e documentare qualsiasi adattamento di formulazione tu richieda.
